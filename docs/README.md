# OctoAcme Project Management Docs

## 概要

このフォルダは、OctoAcme のプロジェクト管理プロセスに関するドキュメントの入口です。新メンバーのオンボーディングや、既存メンバーがプロセスを参照する際のハブとして機能します。各フェーズ・役割・コミュニケーション戦略・品質保証のガイドラインが `docs/` 配下のドキュメントに整理されており、本 README からすべてのドキュメントにアクセスできます。

## プロジェクト管理プロセス概要

OctoAcme のプロジェクト管理は、「顧客価値を最優先に、小さく反復的に届ける」ことを軸にしたライフサイクルで整理されています。全体は **Initiation → Planning → Execution → Release → Close/Retrospective** の流れで進み、Initiation では One-pager（課題・目標・成功指標など）、ステークホルダーとコミュニケーション計画、ハイレベルなマイルストーン、初期リスク、必要な体制を最小成果物として揃えます。成功指標が明確で、優先度とリソースに合意できた時点で Planning に進む「意思決定ゲート」を設け、軽量ながらも承認の筋道を明確にしています。

役割分担は、**Project Manager（PM）** が進行・スケジュール・リスク・コミュニケーションを担い、**Product Manager（PdM）/Product Lead** が成果（アウトカム）定義、優先順位付け、成功指標の測定を担います。**Developers** は設計・実装・テスト・レビューに責任を持ち、必要に応じて **QA/Testing** が受け入れ観点で品質確認を支援します。Planning ではキックオフを実施し、受け入れ基準を含むバックログ（テンプレートあり）を作成・見積りし、Definition of Done（DoD）や QA 方針、依存関係、リリース計画とマイルストーンマップを整備して「実行可能な計画」に落とし込みます。

コミュニケーションは一定のリズムを重視し、週次の PM+PdM 同期や、実行フェーズのデイリースタンドアップ／週次デリバリー同期、マイルストーンごとのデモ・レビューで透明性を確保します。ステークホルダー向けには週次（またはマイルストーン）で状況共有し、単一の真実のソース（例：README やリリース文書）を維持します。リスクはリスクレジスター（Impact/Likelihood/Owner/Mitigation/Status 等）で管理し、エスカレーションは「チーム内トリアージ → PM が Product Lead と依存チームへ → スポンサー級」の段階を明確化、インシデント時の連絡テンプレートやセキュリティインシデントの扱いも定義しています。

品質保証は、日々の実行プロセスとリリース手順に組み込まれています。PR は小さく保ち、Issue リンクと受け入れ基準を明記し、CI でテスト／Lint／セキュリティスキャンを通した上でレビュー（最低 1 承認などチーム方針）を行います。テストはユニット・統合に加え、重要導線の E2E スモークをリリース前に実施し、必要に応じて手動 QA で受け入れを確認します。リリースでは、受け入れ基準達成・CI 合格・リリースノート・ロールバック/緩和策・スモークテスト準備を前提条件とし、ステージング→本番→事後検証→周知のチェックリストでリスクを下げます。最後に、スプリント／リリース／インシデント後のレトロで学びをアクションに変換し、バックログ化して継続改善を回す運用になっています。
